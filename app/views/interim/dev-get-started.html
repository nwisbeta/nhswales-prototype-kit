Overview

The NHS Wales Developer Portal tells users everything they need to know about the APIs that interact with key NHS Wales systems.

We've grouped information about our APIs, how to test them and how to integrate them in one place. All content is written clearly, and it's easy to navigate the site to find what you need.

We're currently in Beta. This means that we're still testing the site and making changes. Give feedback to help us to improve.

Who the portal is for

The site is open and its content can be viewed by anyone. However, the content is currently only intended for:

  • NHS Wales development teams
  • development teams contracted to integrate with NHS Wales systems

As the site is not intended for general use by patients or the public, some content has not been translated into Welsh. Read more about our Welsh standards.

Who maintains the developer portal

The site is maintained by the Digital Health Ecosystem Team Wales (DHEW) programme.

DHEW is a collaboration between Digital Health and Care Wales (DHCW) and the Life Sciences Hub Wales (LSHW), and aims to make it easier and faster to adopt digital healthcare technology in Wales.

If you're interested in partnering with us as a third party developer, come to a DHEW engagement event, or look at our standards and resources to find out more.

1. Learn how NHS Wales use APIs

Our architecture overview

NHS Wales' digital architecture is made up of information management systems and data repositories. Messaging components, including APIs, are used to mediate communication between them.

Information management systems

These provide a user interface for viewing and updating information, such as:

  • GP systems, like Vision
  • hospital patient administration systems, like the Welsh Patient Administration system (WPAS)
  • specialist systems, like the Welsh Laboratory Information System (WLIMS)
  • reference management, like the Welsh Pathology Handbook

Data repository services

These provide secondary or consolidated store of information from other systems, such as:

  • the Welsh Care Record Service (WCRS)
  • the Welsh Results Reporting Service (WRRS)
  • the Welsh Demographics Service (WDS)
  • the Master Patient Index (MPI)
  • the Welsh Reference Data and Terminology Service (WRTS)

Messaging components

These mediate the communication between the various systems and services. In some cases,the components are APIs that allow for direct communication to the underlying system, such as:

  • Welsh GP Record API (WGPR)
  • WPAS APIs
  • WCRS document search and retrieval APIs
  • MPI patient demographics query API

In other cases, they are APIs or message processors separate to the receiving system and typically perform some routing, logging, buffering or transformation of the message flows, such as:

  • NHS Wales Messaging Fabric (Fiorano Enterprise Service Bus)
  • GP Links (Iuvo)
  • Welsh Clinical Communications Gateway (WCCG)

Identity and access management services

These control user access to data. Applications should:

  • authenticate users with NADEX or AC3
  • apply role based Access Controls locally or with AC3
  • WCRS document search and retrieval APIs
  • record usage to local audit logs or to the Central Audit Service (CAS)

Key systems and services

There are multiple key systems and services used within NHS Wales that you can integrate with using APIs.

These include:

  • National Active Directory and Exchange (NADEX),
  • Account control v3 (AC3)
  • Welsh Reference Data Service (WRDS)
  • Terminology service
  • Enterprise Service Bus
  • Welsh Clinical Communication Gateway (WCCG)
  • Requesting services
  • Wales Demographics Service (WDS)
  • Master Patient Index (MPI)
  • Welsh GP Record (WGPR)
  • Welsh Care Record Service (WCRS)
  • Welsh Results Reports Service (WRRS)

Read an overview of key NHS Wales systems and services (opens in a new tab) in the NHS Wales Open Architecture Docs.

2. Learn our API technologies and standards

SOAP/XML

Many of the APIs used to access NHS Wales national systems are Simple Object Access Protocol (SOAP) web services. SOAP APIs use XML to format the message and HTTP to send it. Read more about SOAP specifications (opens in a new tab).

Web Service Description Language (WSDL) documents are used to provide a machine readable definition of SOAP APIs. Read more about WSDL documents (opens in a new tab).

If you need to connect to system using SOAP API, you'll usually use a tool that automatically generates the client code for you based on the WSDL.

Data standards and HL7

The way we structure data is different from system to system, but there are some common standards we use.

The Welsh GP Record API and the Welsh Clinical Communications Gateway (WCCG) use locally defined standards for representing health record information.

Other systems, like the Master Patient Index (MPI) and the Welsh Lab Information Management System (WLIMS), use the HL7 v2 message structures for their APIs.

The MPI API also conforms to the IHE Profile for performing Patient demographic queries and patient identifier cross-reference operations.

FHIR and OpenEHR

We also use HL7 FHIR and OpenEHR, which are modern standards for healthcare systems.

FHIR is a standard for communication between systems. It defines structures for common data entities, for example patient, and an API. The core standard aims to cater for 80% of use cases, with options to extend and customise definitions for other use cases. The terminology service is an example of a system that uses FHIR standards.

OpenEHR is a separate standard that publishes its own data definitions and API specifications for use in healthcare. In addition, it defines models for data persistence in the underlying system. The Clinical Data Repository is an example of a service that uses OpenEHR. Read more in the blog post What is openEHR and why is it important? (opens in a new window).

We expect that all new systems will provide an API that conforms to one of these two standards.

3. Search for APIs

You can find all available APIs in the API catalogue.

APIs are tagged by their state, for example:

  • live APIs, which are already in widespread use
  • experimental APIs, which you can test in the sandbox and suggest improvements

As we're in Beta, we're still developing the technical documentation for some APIs.

4. Test and go live

Sandbox APIs

We have some sandbox APIs available for users to test, learn, and start to integrate with.

To test an API in the sandbox, you should:

  1. Sign in or register an account.
  2. Subscribe to the sandbox.
  3. Search our sandbox APIs to find the one you'd like to try.

Testing and production environments

If you're ready to integrate an API with your application, you can request access to:

  • system integration testing (SIT), to validate an application against integration testing requirements
  • user acceptance testing (UAT), for application end users to test a new feature in a non-production environment (if required)
  • production environment, for pilot and go-live of your application

Access to these environments is subject to an assurance process.

The assurance process

We will ask you some questions to make sure that your use case for integrating your application with an API meets with our standards.

This process can change depending upon which API you're requesting access to. Find the API you'd like to access in the API catalogue and read its documentation to see the most up to date process.

If the API does not have information about its assurance process, email ecosystem@wales.nhs.uk.

Production access will be provided once your application passes our checks.

5. Get help and support

For general support with the NHS Wales Developer Portal, email ecosystem@wales.nhs.uk.

If you need help with an API, check the support section of the API's documentation for more specific information about how to get support. This makes sure you are put in touch with the right team more quickly.

Welsh language standards

What content we provide in Welsh

We aim to provide Welsh translation where possible. However, because the content of this website is not intended for patients or the public, we have not translated more technical information.

The content that is translated into Welsh includes:

  • the homepage
  • get started
  • site navigation

Welsh translation if you're applying for a tender

If you're applying for a tender, you can ask us to translate any of this site's content into Welsh.

Email ecosystem@wales.nhs.uk and include:

  • your name and organisation
  • which tender you are applying for
  • the url of each page you need in Welsh

Your Welsh language requirements when creating apps for patients and the public

You have a responsibility to consider Welsh language requirements when creating an application for use by Welsh public, patients and clinicians.

This ensures that the Welsh language is treated no less favourably than English, and allows all users to access information in Welsh if they choose to.

There is lots of clear advice available in the Welsh Language Commissioner's Welsh Language Considerations document (opens in a new window). It includes information about bilingually designing your application's:

  • architecture
  • user interface
  • language selection
  • locales, alphabets and character sets
  • data management
  • application outputs
  • addresses and geographical information

Another helpful resource is the bilingual technology toolkit for good user experience (opens in a new window). Use this to break down your application's Welsh language requirements by priority.
