About the developer portal

This site is for application developers working on integrations with and within NHS Wales.

Here you'll find technical guidance on how to interact with key systems using APIs.

Such information can be hard to obtain as it mostly spread across separate documents, in different formats, and often only available on request.

Our aim is to make that information available within a single "Developer Portal", in a format that's easy to read, navigate and find via an internet search.

This site is currently in "Beta", which means things might change (and/or break) but it also means we'd love your feedback, so feel free to get in touch [link to feedback form?].

The site is maintained by a team at Digital Health and Care Wales as part of the Digital Health Ecosystem Wales programme.

Although anybody is free to browse the content on this site, it is currently only intended for:

  • NHS Wales development teams
  • Development teams contracted to integrate with NHS Wales systems

If you're interested in how you can partner with NHS Wales, then the team at Digital Health Ecosystem Wales provide resources and events that you may find useful.

Our API technologies and standards

SOAP/XML

The APIs used to access many NHS Wales national systems are SOAP Web Services.

SOAP APIs use XML to format the message and HTTP to send it.

Ref: https://www.w3.org/TR/soap/

WSDL documents are used to provide a machine readable definition of SOAP APIs

https://www.w3.org/TR/2001/NOTE-wsdl-20010315

If you need to connect to system using SOAP API, you'll usually use a tool that automatically generates the client code for you based on the WSDL.

Data Standards and HL7

The way that data is structured differs between systems, but there are some common standards in use.

The Welsh GP Record API and the Welsh Clinical Communications Gateway (WCCG) use locally defined standards for representing health record information.

Other systems, like the Master Patient Index and the Welsh Lab Information Management System (WLIMS) use the HL7 v2 message structures for their APIs.

The Master Patient Index API also conforms to the IHE Profile for performing Patient Demographic Queries and Patient Identifier Cross-Reference operations.

FHIR and OpenEHR

HL7 FHIR and OpenEHR are modern standards for healthcare systems.

FHIR is a standard for communication between systems. It defines structures for common data entities (e.g Patient, CarePlan) and an API. The core standard aims to cater for 80% of use cases, with options to extend and customise definitions for other use cases.

OpenEHR is a separate initiative that publishes their own data definitions and several API specifications for use in healthcare, but additionally define models for data persistence in the underlying system.

If it helps you can think of FHIR a bit like a standard for the USB port (so you can easily plug in anything you want to), but OpenEHR is more like a standard for how the whole computer should work (so you can easily add an extra RAM stick or a video card AND plug things in to the Thunderbolt port).

A few systems in NHS Wales use these standards, e.g. Terminology Server (FHIR), new Clinical Data Repository (OpenEHR) and you can expect that all new systems are likely to provide an API that conforms to one of these two standards.

Try APIs in the sandbox

Alongside the technical documentation, we also provide sandbox versions of some key APIs. Developers can use the sandbox environment to "play around" with the APIs, learn how to use them and start developing an integration.

You can request access to the Sandbox APIs using this site [subscribe to sandbox link]

Find out more about requesting access to APIs.

*to move* Requesting access to APIs

After trying features in the sandbox, you can request access to:

  • system integration testing (SIT), to validate an application against integration testing requirements
  • user acceptance testing (UAT), for application end users to test a new feature in a non-production environment (if required)
  • production environment, for pilot and go-live of your application

The assurance process

This is a series of checks to make sure that your use case for the API meets with our standards.

The assurance process depends upon the API you're requesting access to. Find the API you'd like to access in the API catalogue and read its technical documentation to learn more.

If the API does not have information about its assurance process, contact us by email.

Production access will be provided once your application passes assurance checks.
